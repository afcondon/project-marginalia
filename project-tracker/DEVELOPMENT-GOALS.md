# Project Tracker: Development Goals

## What This Is Becoming

A **keyboard-first dispatch board and navigator** for all projects across programming, music, house, woodworking, garden, and infrastructure. Not a project management app in the Jira sense — a rapid-capture, rapid-context-switch tool that sits between thinking and doing.

The server architecture (HTTPurple + DuckDB, accessible via TailScale) means both humans and agents operate on the same data. The tracker is the shared brain.

## Two Modes

### 1. Keyboard-Driven Browser App

The power tool. Fully keyboard navigable, not dependent on any particular laptop.

Interaction model:
- Arrow keys / j/k to move through cards
- `/` to search (fuzzy, across all projects)
- `e` to edit, `Esc` to close
- Number keys 1-7 for traffic light status changes
- `n` to add a note
- `d` to dispatch (tee up agent work)
- Command palette for less frequent actions

The quick-capture loop: search (3 chars) -> find project -> add note/feature -> move on. 3-4 keystrokes, not 3-4 clicks.

### 2. iOS / Apple Watch App

The capture tool. Minimal UI, optimised for on-the-go input.

**iOS**: Search box + project detail + dictation button + camera button. Find a project, dictate a note, attach a photo. Talks to the same API via TailScale.

**Apple Watch**: Raise wrist, say "note on greenhouse project: ordered the shade fabric", done. Complication for quick access.

Architecture: both talk to the same HTTPurple backend. The agent API (`/api/agent/*`) is already close to what these apps need — compact JSON, simple CRUD.

## Navigator, Not Owner

The tracker should **index and navigate to** documentation, not duplicate it. Each project card becomes a portal to:

- Design docs and architecture files (e.g. `ShapedSteer/ARCHITECTURE.md`)
- KB articles (`purescript-polyglot/docs/kb/*`)
- Worklog entries (`purescript-polyglot/docs/worklog/*`)
- Plan documents (`purescript-polyglot/docs/kb/plans/*`)
- Session artifacts generated by Claude during work
- Image attachments (already linked from infovore-larder-db)

These are **references** — the tracker stores paths/URLs, not the content itself. The canonical location of each document stays where it is (in its project repo, in the KB, etc.).

### What this means for the data model

The `attachments` table already supports this (it has `file_path`, `mime_type`, and `description`). We'd extend it or add a parallel `documents` table:

```
project_documents
  project_id   -> projects.id
  title        text         -- "Architecture doc", "Release plan", "Session log 2026-04-06"
  doc_type     text         -- plan | architecture | worklog | session | reference | brainstorm
  file_path    text         -- canonical location on disk
  url          text         -- alternative: web URL
  status       text         -- draft | active | implementing | done | superseded
  superseded_by integer     -- links to the replacing document
  created_at   timestamp
  updated_at   timestamp
```

## Document Lifecycle

Design docs and plans have their own progression, parallel to but distinct from the project's status:

```
brainstorm -> draft -> active -> implementing -> done
                                     |
                                     +-> superseded (-> superseded_by)
```

This is the cycle currently managed via A5 booklet printouts. The tracker could:
- Show which docs are current vs stale for each project
- Surface docs that need review ("this plan is 3 months old and still 'active'")
- Generate printable summaries (preserve the A5 booklet workflow but from live data)
- Track the relationship between a plan and its implementation status

### The documentation Claude generates

Every substantive Claude session produces artifacts: worklogs, design decisions, code changes, plans. These currently land in `purescript-polyglot/docs/worklog/YYYY-MM-DD.md` and scattered KB articles. The tracker could:

1. **Index them**: scan worklog/KB directories periodically, match to projects by content or explicit tags
2. **Link them**: when a session log mentions a project, the project card shows the link
3. **Track freshness**: flag when a plan doc is out of date relative to the code it describes

This is read-only integration — the tracker doesn't write to the KB, it reads and indexes.

## Dispatch: From Idea to Agent Work

A note on a project can be just a note ("fix the gate hinge") or an **intent** that's actionable by an agent. The difference:

- **Note**: freeform text, human context, no machine action
- **Task**: structured enough to be worked on (by human or agent)
- **Agent task**: has enough context to start a Claude session — project path, description, constraints, target files

### Dispatch flow

```
Human captures idea on project
  -> Refines into task (adds detail, constraints)
  -> Marks as "dispatch-ready"
  -> Dispatch triggers agent work:
       - Sets working directory to project's source_path
       - Passes task description as prompt
       - Agent works, writes results back to project notes
       - Agent updates task status
  -> Human reviews on next visit to project card
```

For non-programming projects, "dispatch" means something different:
- Generate a materials list
- Research options and summarise
- Create a shopping list
- Draft a communication

### The return path

When an agent finishes, it writes back via `POST /api/agent/projects/:id/notes`. The project card shows agent-authored notes alongside human ones, with clear attribution (`author` field distinguishes `human`, `architect`, `implementer`, etc.).

The `agent_sessions` table (already in the schema) tracks what ran, when, and what it accomplished.

## Integration Points

| System | Role | Direction |
|--------|------|-----------|
| infovore-larder-db | Source data (projects, notes, images) | Import |
| purescript-polyglot/docs/kb/ | Design docs, plans | Read/index |
| purescript-polyglot/docs/worklog/ | Session logs | Read/index |
| Project repos (ShapedSteer, etc.) | Source code, ARCHITECTURE.md | Read/navigate |
| Claude Code CLI | Agent dispatch | Outbound |
| Agent API | Agent read/write during sessions | Bidirectional |
| TailScale | Network access from any device | Transport |
| Crucial4TB (MacMini) | Image/file storage | Read |

## What's Built So Far (2026-04-06)

- PureScript/Halogen frontend with filter pills, traffic light status dots, slide-out detail, keyboard-ready structure
- HTTPurple server with full CRUD + agent-optimised API + dependency CRUD
- DuckDB backend with 120 projects, 14 tags, 41 linked images
- Hash-based routing, auto-save on edit
- Accessible via TailScale (server runs on MacMini)

## Next Steps (Rough Priority)

1. **Keyboard navigation** — j/k movement, `/` search, number keys for status, `n` for notes
2. **Image display** — serve attachments via the API, show thumbnails on cards and in detail panel
3. **Document references** — add `project_documents` table, scan and index existing docs
4. **Dispatch prototype** — simple version: note marked as agent-task generates a `claude` CLI invocation
5. **iOS app** — SwiftUI, TailScale, dictation + camera
6. **Document lifecycle tracking** — status on plans/docs, freshness alerts
7. **Print/export** — A5 booklet-compatible summaries
